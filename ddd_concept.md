# DDD (Domain Driven Design)

업무 규칙을 팀의 언어로 경계 안에 모아두는 것
- 업무 규칙 = 도메인 규칙 (Business Rules)
- 팀의 언어 = 개발자 + 기획자 + 현업이 “같은 뜻으로” 쓰는 말
- 업무 규칙은 특정한 경계(Bounded Context) 안에만 존재해야 한다

## Bounded Context
- **하나의 소프트웨어를 여러 모듈의 조합으로 나누는 것**
- 모듈은 최대한 독립적이고 폐쇄적으로 만든다.

## Ubiquitous Language
- **같은 개념에 이름 2개 이상 붙이지 않기**
- 예: 상품을 Goods, Product, Item 등으로 섞어서 부르지 않기
- 현업(실무담당자)이 사용하는 언어를 사용해야 한다.
- 현업에서 상품을 Product라고 부르면, 코드에서도 Product를 사용해야 한다. (Goods, Item 등 사용 금지)

## 규칙: 한 곳에 모으기
- **도메인 규칙은 한 곳에 모아둬야 한다.**
- 예: 상품 가격 할인 규칙
  - 상품 가격 할인 규칙이 여러 군데 흩어져 있으면 안 된다.
  - 상품 가격 할인 규칙이 바뀌면, 한 군데만 수정하면 된다.

## 비유
- ❌ DDD 없음 → 회사 규칙이 단톡방, 메일, 구두, 엑셀에 흩어져 있음
- ✅ DDD 있음 → “이 규칙은 인사팀 규정집 3장 2조”라고 딱 정해져 있음

## DDD의 목표
- 업무 변경이 발생했을 때 코드의 어느 부분을 바꿔야 하는지 바로 알 수 있는 구조를 만든다.
